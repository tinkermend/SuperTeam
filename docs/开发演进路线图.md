# SuperTeam 开发演进路线图

日期：2026-05-27

## 依据

本文基于以下文件整理：

- `AGENTS.md`
- `docs/数字员工平台技术方案.md`
- `docs/设计想法.md`

路线图只定义阶段目标、核心工作和验证逻辑，不提前冻结具体接口字段、数据库表结构和任务状态枚举。

## 总体原则

- 第一阶段就引入 Temporal，但只用于必要的长流程、暂停恢复、重试、超时和补偿。
- Control Plane 使用 Go + chi/net/http，以 REST/OpenAPI 为主。
- Runtime Agent 是部署在服务器节点上的 Go daemon，只负责节点执行、租约、Provider 会话、日志和工件回传。
- 第一阶段只实现 Web 控制台主链路。Desktop 仅保留空壳或占位，暂不做业务适配；Web 主链路完整后再实现桌面壳、通知和快速查看。
- 外部能力通过 Capability 注册、授权、HTTP 调用和审计接入，不把客户差异写进核心流程。
- MVP 先跑通固定链路：需求分析 -> 人类确认 -> Runtime Agent 执行 -> 回传结果和工件 -> 人类验收。

## 第一步：建立工程骨架和契约基线

目标：先把仓库结构、应用边界和 API 契约跑起来，避免后续实现混在一起。

核心工作：

- 建立 `apps/web`、`apps/control-plane`、`apps/runtime-agent`；`apps/desktop` 仅保留空壳或占位。
- 建立 `packages/ui`、`packages/views`、`packages/core`、`packages/api-client`。
- 建立 `contracts/control-plane`、`contracts/runtime`、`contracts/provider`。
- Control Plane 提供基础 health API。
- Web 通过生成的 API client 读取 health API。
- Runtime Agent 能读取配置并启动空 daemon。
- 建立基础开发命令、测试命令和目录约束。

验证方式：

- `go test ./...` 能通过当前 Go 骨架测试。
- 前端 typecheck 能通过。
- Control Plane 启动后，`GET /health` 返回正常。
- Web 页面能显示 Control Plane 健康状态。
- Runtime Agent 启动后能输出节点启动日志，但不执行任何任务。

预期效果：

- 仓库边界清晰。
- Console、Control Plane、Runtime Agent、Contracts 不互相侵入。
- 后续每个阶段都有明确落点。

## 第二步：实现 Control Plane 最小闭环底座

目标：让控制平面具备任务、审批、工件、审计和 Temporal 工作流的最小能力。

核心工作：

- 建立 Control Plane 的领域包：`task`、`workflow`、`approval`、`artifact`、`audit`、`runtime`、`storage`、`config`。
- 接入 PostgreSQL、Redis、S3 兼容存储的基础封装。
- 建立 OpenAPI 契约和 `oapi-codegen` 生成流程。
- 建立最小任务创建 API。
- 建立最小 HumanDecisionRequest 记录能力。
- 建立最小 Artifact 记录能力。
- 建立最小审计事件记录能力。
- 建立 Temporal worker，并跑通一个可暂停、可恢复的最小 workflow。

验证方式：

- 创建一个任务后，数据库中能看到任务记录。
- 提交一个人工确认请求后，数据库中能看到确认记录和审计记录。
- 上传或记录一个工件后，能通过 API 查询到。
- Temporal workflow 能在人工确认点暂停，并在提交确认后继续。
- 控制平面 API 的 OpenAPI 文件能生成客户端，生成结果被 Web 正常调用。

预期效果：

- 控制平面开始成为系统事实来源。
- 人类确认、工件、审计不再只是聊天文本。
- Temporal 的暂停恢复能力在最小范围内被验证。

## 第三步：实现 Runtime Agent 注册、心跳和任务租约

目标：让部署在节点上的 Runtime Agent 能被控制平面管理，但还不急着接真实 Provider。

核心工作：

- Runtime Agent 实现 `daemon`、`controlplane`、`lease`、`slots`、`events`、`config`、`health`。
- Control Plane 提供 Runtime 注册、心跳、任务 claim、lease renew API。
- Runtime Agent 定期注册和心跳。
- Runtime Agent 主动 claim 可执行任务。
- Runtime Agent 维护本机执行槽位。
- Runtime Agent 通过 WebSocket 回传节点事件。

验证方式：

- 启动 Control Plane 和 Runtime Agent 后，控制平面能看到节点在线。
- 停止 Runtime Agent 后，控制平面能识别心跳中断。
- 创建一条可执行测试任务后，Runtime Agent 能 claim 到任务。
- Runtime Agent 能续租任务，避免同一任务被重复领取。
- Web 控制台能看到 Runtime Agent 的注册、心跳和事件回传。

预期效果：

- Control Plane 和 Runtime Agent 的职责边界被跑通。
- 系统具备节点管理、任务领取和租约续期的基础能力。
- 后续接入 Provider 时，不需要改控制平面的核心调度方式。

## 第四步：接入第一个 Provider Adapter

目标：让 Runtime Agent 能通过统一 provider contract 管理一个真实执行器，优先 Claude Code 或 OpenCode。

核心工作：

- 定义 `contracts/provider` 中的输入、事件、结果、工件契约。
- Runtime Agent 实现 `providers` 包。
- 先实现 mock provider，用于稳定测试事件和工件链路。
- 再接入一个真实 Provider Adapter。
- 当 Provider 协议不完整时，使用 CLI、PTY 或 HTTP adapter 兜底。
- Provider 执行过程中的日志、事件、结果和工件统一回传 Control Plane。

验证方式：

- mock provider 能完成一条测试任务，并生成固定 ExecutionResult。
- 真实 Provider 能在受控工作目录中执行一个小型代码任务。
- 执行日志能实时出现在控制平面事件流中。
- 执行结束后能查询到 ExecutionResult 和 Artifact。
- Provider 失败时，能记录失败原因和可审计事件。

预期效果：

- 平台不绑定某个 Provider 的内部实现。
- Runtime Agent 具备管理本机 Provider 会话的能力。
- 真实代码任务可以进入平台闭环。

## 第五步：打通 MVP 人类确认和验收链路

目标：跑通第一条完整可控闭环。

核心工作：

- Web 提供需求输入、任务详情、人工确认、执行日志、工件查看和人类验收页面。
- Control Plane 将需求分析、人工确认、Runtime 执行、工件回传、人类验收串成 Temporal workflow。
- 数字员工输出结构化 Handoff、Finding、DecisionRequest 和 Artifact。
- Runtime Agent 执行结果回传后，任务进入人类验收环节。
- 所有关键动作写入 Audit。

验证方式：

- 人类输入一个小型代码需求。
- 系统生成结构化需求分析和确认请求。
- 人类确认后，Runtime Agent 领取任务并调用 Provider。
- Provider 执行后回传日志、结果和工件。
- 人类能在 Web 上查看完整过程并提交验收结论。
- 审计记录能串起需求输入、确认、执行、结果、验收全过程。

预期效果：

- MVP 的核心价值成立：可控、可暂停、可恢复、可审计。
- 平台不再只是任务列表，而是能驱动真实执行闭环。
- 后续增加数字员工、连接器和治理能力时，有稳定主链路。

## 第六步：稳定 Web 共享页面边界

目标：在 MVP 主链路稳定后，把可复用的 Web 业务页面和领域逻辑沉淀到共享包，为未来 Desktop 做准备，但仍不开发 Desktop 业务功能。

核心工作：

- 将任务列表、任务详情、审批面板、工件查看、执行日志等稳定页面沉淀到 `packages/views`。
- 将 API hooks、schema、权限判断和领域状态沉淀到 `packages/core`。
- 保持 `packages/views` 不直接依赖 `next/navigation`、Next Server Actions 或桌面端路由。
- Web 继续作为唯一业务入口，Desktop 只保留空壳或占位。

验证方式：

- Web 功能在抽取到共享包后行为不变。
- 共享页面可以通过 props、hooks 和平台适配层运行，不依赖 Next 专属 API。
- 新增业务页面时，默认先判断是否应进入 `packages/views`。
- `apps/web` 只负责路由、布局挂载和平台适配。

预期效果：

- Web 主链路保持开发效率。
- 未来 Desktop 可以复用稳定后的业务页面。
- 避免第一阶段为 Desktop 适配过早付出复杂度。

## 第七步：扩展多数字员工协作

目标：从固定 MVP 链路扩展到需求、开发、测试等多数字员工协作。

核心工作：

- 引入 Coordinator。
- 固化 Handoff、Finding、Blocker、ExecutionResult、Risk、NextActionProposal 等结构化对象。
- 增加测试数字员工。
- 支持流程模板版本管理。
- 支持上下文切片注入。
- 增加 Provider 选择策略。

验证方式：

- 一个需求能经过需求分析、开发执行、测试结论三个阶段。
- 每个阶段都有明确输入、输出和交接包。
- Coordinator 能根据结构化对象决定下一步。
- 数字员工之间不直接自由聊天。
- 人类能查看每个阶段的判断依据。

预期效果：

- 平台从单链路闭环进入多角色协作。
- 结构化交接成为协作事实来源。
- 后续治理、评分、自动路由有数据基础。

## 第八步：接入外部能力和客户差异

目标：让不同客户的系统和流程差异通过配置、连接器和能力注册解决，而不是进入核心代码。

核心工作：

- 建立 Tenant Profile。
- 建立 Connector 框架。
- 建立 Semantic Mapping。
- 建立 Capability 注册、授权、调用和审计。
- 支持客户级流程模板。
- 支持客户级审批策略。

验证方式：

- 接入一个通用 HTTP 外部能力。
- 同一类业务对象能通过 Semantic Mapping 适配两个不同来源。
- 数字员工只读取平台统一对象，不直接理解客户系统字段。
- 外部能力调用有授权记录和审计记录。

预期效果：

- 客户差异被隔离在边界层。
- 平台核心不因接入客户而改代码。
- 后续多租户治理具备基础。

## 第九步：企业级治理和规模化

目标：让平台从可用进入可治理、可运营、可规模化。

核心工作：

- 完整权限体系，逐步接入 OpenFGA。
- 审批流升级。
- 操作风险分级。
- 审计报表。
- Runtime Agent 资源治理。
- 多 Provider 策略。
- 成本统计。
- 失败重试和补偿策略完善。
- 安全策略和密钥治理。

验证方式：

- 不同角色看到不同任务、审批和能力。
- 高风险操作必须进入审批。
- Runtime Agent 并发、租约、失败重试可控。
- 审计报表能追踪关键决策和执行结果。
- 多 Provider 策略能根据任务或租户选择执行器。

预期效果：

- 平台具备企业生产使用基础。
- AI 执行风险可控。
- 团队能基于审计和成本数据持续治理平台。

## 第十步：补齐 Desktop 原生壳和通知

目标：在 Web 控制台和共享页面稳定后，让桌面端成为同一控制台的原生入口。

核心工作：

- 使用 Tauri 承载桌面壳、通知中心和快速查看。
- 复用 `packages/views` 的主要页面。
- Desktop 接入平台适配层，不直接依赖 Web 路由。
- Desktop 提供登录、任务查看、审批通知、快速打开任务详情。
- Desktop 不承担本机执行能力，不管理本机 Provider。

验证方式：

- Web 和 Desktop 能看到同一批任务、审批和工件。
- Web 已稳定的共享页面不需要在 Desktop 重写业务逻辑。
- 新审批产生后，Desktop 能发出原生通知。
- 点击通知能进入对应任务详情。

预期效果：

- Desktop 成为控制台的便捷入口，而不是第二套产品。
- Web/Desktop 共享逻辑被验证。
- 桌面端带来通知和快速访问价值，但不改变控制平面核心架构。

## 里程碑验收标准

第一阶段完成时，必须能验证：

- Web 能创建需求任务。
- Control Plane 能记录任务、人工确认、工件和审计。
- Temporal 能暂停并恢复一个最小工作流。
- Runtime Agent 能注册、心跳、claim、续租和回传事件。
- 一个 Provider Adapter 能完成真实或受控的代码任务。
- 人类能在 Web 上验收结果。

第二阶段完成时，必须能验证：

- 多数字员工通过结构化对象协作。
- 每个阶段都有可查询的输入、输出和交接包。
- Coordinator 能基于结构化结果推进流程。

第三阶段完成时，必须能验证：

- 外部能力通过 Connector 和 Capability 注册接入。
- 客户差异通过 Tenant Profile、Semantic Mapping 和 Policy 隔离。
- 平台核心不需要为客户写硬编码逻辑。
