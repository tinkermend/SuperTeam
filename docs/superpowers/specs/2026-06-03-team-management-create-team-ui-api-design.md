# 团队管理与新建团队闭环优化 Spec

> 日期：2026-06-03
> 状态：待书面评审
> 决策：采用“团队管理列表页 + 右侧两步新建团队抽屉 + 真实前后端接口闭环”的优化方案

## 1. 背景

团队管理页当前已经具备基础列表、团队详情框架和部分真实接口，但实际页面体验与既定设计图存在明显差距：列表缺少完整筛选工具条、主操作层级、负责人可读摘要、语义状态和行操作；新建团队流程尚未按右侧两步抽屉落地；部分交互如果只靠前端拼接或 mock 数据，会继续造成“界面像管理台但数据流不真实”的问题。

本 spec 聚焦一次纠偏：让团队管理页面、新建团队页面和前后端接口能力形成真实闭环。实现时必须使用本地真实 Control Plane API 和真实数据库数据，不以 mock 数据作为完成证据。

## 2. 目标

- 团队管理列表页按原设计恢复控制台信息架构：标题区、筛选工具条、主操作、表格、状态、负责人和行操作。
- 新建团队采用右侧抽屉式两步流程：基础信息、初始成员。
- 负责人和初始成员均从已有租户用户中搜索选择，不能手输 UUID。
- 创建团队在第二步提交时一次性落库，避免产生半成品团队。
- 后端补齐列表筛选、负责人摘要、用户搜索和一次性创建团队加成员的事务能力。
- 前端所有列表、搜索、创建和刷新均走真实 API，不走 mock。
- 测试和烟测覆盖前端、后端、接口契约和真实登录后的页面验证。

## 3. 非目标

- 不在新建团队抽屉里创建新用户、邀请用户或开户。
- 不在本阶段重做团队详情 6 个 Tab 的完整 UX。
- 不实现治理配置草稿编辑器、能力绑定和审批流程的深度改版。
- 不实现团队硬删除。
- 不暴露 OpenFGA tuple、授权 DSL 或底层权限模型给业务用户。
- 不把新建团队流程扩展为 `/teams/new` 独立页面。

## 4. 页面结构

### 4.1 团队管理列表页

`/teams` 保持企业控制台的高信息密度，不做营销式大卡片。首屏结构为：

- 顶部 Shell：沿用现有控制台顶部栏和全局搜索。
- 页面标题区：左侧展示“团队管理”和说明文案，右侧展示主按钮“新建团队”。
- 筛选工具条：包含搜索、团队状态、治理状态和重置按钮。
- 数据表格：展示团队组织事实、治理摘要和可操作入口。

筛选工具条字段：

- 搜索：匹配团队名称、slug、负责人姓名、负责人邮箱。
- 团队状态：全部、活跃、已禁用、已归档。
- 治理状态：全部、未配置、草案待批准、已生效、需更新。
- 重置：清空搜索与筛选条件，重新请求列表。

表格列：

- 团队：语义图标、名称、slug。
- 状态：`active`、`disabled`、`archived` 的语义 Badge。
- 负责人：头像或头像占位、姓名、邮箱摘要。
- 成员。
- 数字员工。
- 能力。
- 治理状态。
- 当前版本。
- 待批准。
- 更新时间。
- 行操作：查看、编辑、禁用、归档、恢复等，按后端授权结果展示。

视觉规则：

- 继续遵循 `DESIGN.md` 的浅色液态玻璃控制台方向。
- 表格区域使用稳定白色面板、细边框和浅阴影，不能用强透明影响阅读。
- 行 hover 和新建成功后的临时高亮使用浅青绿色背景，不改变行高和列宽。
- 状态色不能只靠颜色表达，必须保留中文文字。
- 空状态、加载态和错误态都留在表格区域内，不挤掉标题和筛选工具条。

## 5. 新建团队抽屉

### 5.1 交互容器

新建团队使用右侧抽屉，不使用独立页面。桌面宽度约 `440px` 到 `520px`，移动端全宽。抽屉顶部展示标题“新建团队”和关闭按钮，中部为步骤内容，底部固定操作区。

步骤：

1. 基础信息
2. 初始成员

底部操作：

- 第一步：取消、下一步。
- 第二步：取消、上一步、创建团队。

取消时如果表单或成员选择已有改动，需要二次确认，避免误关丢失输入。

### 5.2 第一步：基础信息

字段：

- 团队名称：必填。
- 团队标识 slug：必填，支持从团队名称自动建议，用户可编辑。
- 负责人：必填，从已有租户 active 用户中搜索选择。

校验：

- 团队名称不能为空。
- slug 只能使用后端允许的格式，前端做即时格式提示，后端做最终校验。
- slug 唯一性由后端校验；前端可做异步提示，但不能代替提交校验。
- 负责人必须是同租户 active 用户。

第一步只保存本地表单状态，不创建团队。

### 5.3 第二步：初始成员

区域：

- 信息提示：负责人、管理员、审批人需创建后发起特权角色申请。
- 用户搜索：从已有租户 active 用户中搜索。
- 角色筛选：全部、普通成员、只读观察者。
- 候选用户列表：展示姓名、邮箱、账号状态和可选角色。
- 已选择成员列表：展示已选用户、分配角色和移除操作。

角色规则：

- 初始成员只允许 `member` 和 `viewer`。
- 第一步选择的负责人自动成为团队负责人，不在第二步重复添加。
- 如果候选列表出现负责人，前端标记为“负责人”并禁选。
- 同一用户不能重复加入。
- 第二步不能直接添加 `owner`、`admin`、`approver`。

提交规则：

- 点击“创建团队”时，一次性提交基础信息和初始成员。
- 提交期间禁用重复提交。
- 成功后关闭抽屉、刷新列表，并临时高亮新建团队行。
- 失败时保留基础信息、成员选择和当前步骤。

## 6. 接口契约

### 6.1 团队列表

扩展现有接口：

```text
GET /api/v1/teams?q=&status=&governance_status=&limit=&offset=
```

查询参数：

- `q`：团队名称、slug、负责人姓名、负责人邮箱的模糊搜索。
- `status`：`active`、`disabled`、`archived`。
- `governance_status`：`not_configured`、`draft_pending`、`active`、`needs_update`。
- `limit`、`offset`：分页参数。

响应字段至少包含：

- 团队基础信息：`id`、`tenant_id`、`name`、`slug`、`status`、`created_at`、`updated_at`。
- `human_owner`：`user_id`、`username`、`display_name`、`email`、`status`。
- `member_count`。
- `digital_employee_count`。
- `capability_count`。
- `governance_status`。
- `current_revision`。
- `pending_draft_count`。
- `risk_summary`。
- `allowed_actions` 或行级可操作摘要。

后端负责筛选和聚合，前端不能先拉全量再本地过滤。

### 6.2 租户用户搜索

第一版复用用户管理接口，并补齐搜索和 active 筛选能力：

```text
GET /api/auth/users?q=&status=active&limit=&offset=
```

用途：

- 负责人选择。
- 初始成员候选列表。

返回字段需要满足抽屉展示：

- `id`。
- `username`。
- `display_name`。
- `email`。
- `status`。

本阶段不新增单独的团队候选用户接口，避免用户搜索能力分叉。团队创建服务端仍必须重新校验用户的租户边界和 active 状态。

### 6.3 创建团队

扩展现有接口：

```text
POST /api/v1/teams
```

请求体：

```json
{
  "name": "安全团队",
  "slug": "security",
  "human_owner_user_id": "00000000-0000-0000-0000-000000000000",
  "initial_members": [
    {
      "user_id": "00000000-0000-0000-0000-000000000001",
      "role": "member"
    },
    {
      "user_id": "00000000-0000-0000-0000-000000000002",
      "role": "viewer"
    }
  ]
}
```

服务端必须在同一事务内完成：

- 校验当前用户具备 `team.create`。
- 校验 slug 唯一。
- 校验负责人存在、同租户、active。
- 校验初始成员存在、同租户、active。
- 校验初始成员角色只允许 `member` 和 `viewer`。
- 去重初始成员。
- 阻止负责人作为普通初始成员重复加入。
- 创建团队。
- 创建负责人 membership，角色为 `owner`。
- 创建初始成员 memberships。
- 写入团队创建和成员添加审计。

成功响应返回 `TeamOverview`，用于前端获得新团队 ID、负责人摘要和当前用户可操作信息。前端成功后仍重新请求团队列表，确保表格聚合字段来自列表事实源。

## 7. 前端数据流

- `TeamsView` 持有筛选状态，并把筛选参数传给 `listTeamSummaries`。
- `TeamListTable` 只负责渲染数据、空状态、错误态和行操作，不负责请求。
- 新建团队抽屉作为独立 feature 组件，内部管理两步表单状态和已选成员状态。
- 负责人选择和成员候选搜索通过 TanStack Query 调用真实用户接口。
- 创建提交通过 TanStack Mutation 调用 `createTeam`。
- 创建成功后 invalidate 团队列表 query，并高亮返回的新团队。
- 创建失败不清空表单。

组件边界：

- `TeamManagementToolbar`：搜索、筛选、重置。
- `CreateTeamDrawer`：抽屉外壳和步骤状态。
- `CreateTeamBasicStep`：基础信息表单。
- `CreateTeamMembersStep`：候选用户、角色选择、已选成员。
- `TeamOwnerCombobox`：负责人搜索选择。
- `TenantUserPicker`：成员搜索和选择。

这些组件优先放在 `apps/web/src/features/teams/components/`。只有跨页面复用后，再沉淀到项目级公共组件。

## 8. 错误处理与状态反馈

列表：

- 加载中：表格区域显示 skeleton 或加载提示。
- 加载失败：表格区域显示错误态和重试入口。
- 无数据：显示“暂无团队”和新建入口。
- 筛选无结果：显示无结果文案和重置入口。

抽屉：

- 字段错误显示在字段下方。
- slug 冲突显示在 slug 字段下方。
- 负责人不可用显示在负责人选择器下方。
- 成员重复、失效或角色非法显示在成员区域。
- 权限不足或事务提交失败显示在底部提交区，并保留当前输入。
- 提交中禁用关闭以外的重复提交动作；关闭时如有未保存更改，需要确认。

权限：

- 没有 `team.create` 时，“新建团队”隐藏或禁用，并通过 tooltip 或禁用原因解释。
- 前端权限展示只做体验优化；后端仍必须执行最终授权、租户边界和事务校验。

## 9. 测试与验收

### 9.1 后端测试

Go 测试覆盖：

- `GET /api/v1/teams` 的 `q`、`status`、`governance_status` 筛选。
- 列表响应包含负责人摘要和聚合计数。
- `POST /api/v1/teams` 一次性创建团队、负责人 membership 和初始成员 memberships。
- slug 冲突失败。
- 负责人不存在、非 active 或跨租户失败。
- 初始成员不存在、非 active 或跨租户失败。
- 初始成员角色为 `owner`、`admin`、`approver` 时失败。
- 负责人被重复放入 `initial_members` 时失败。
- 事务失败时不留下半成品团队。
- 创建成功写入审计记录。

### 9.2 前端测试

Vitest 覆盖：

- `listTeamSummaries` 正确拼接筛选参数。
- `createTeam` 正确发送 `initial_members`。
- 团队列表筛选变更会触发真实 API query key 变化。
- 新建团队抽屉第一步校验。
- 负责人选择只能来自用户搜索结果。
- 第二步只能选择 `member` 和 `viewer`。
- 负责人在候选列表中禁选。
- 提交失败时保留表单和成员选择。
- 创建成功后触发列表刷新。

### 9.3 真实环境烟测

本地启动 Web 和 Control Plane，使用真实数据库完成：

1. 登录平台。
2. 打开 `/teams`。
3. 验证列表筛选、负责人显示和治理状态显示。
4. 打开“新建团队”抽屉。
5. 选择负责人和初始成员。
6. 提交创建团队。
7. 验证列表刷新并显示新团队。
8. 直接查询数据库或调用 API 验证团队、负责人 membership、初始成员 membership 和审计记录存在。

完成证据需要包含真实登录后的页面截图和接口/测试输出，不接受 mock 截图。

## 10. 实施顺序

1. 后端补齐用户搜索、团队列表筛选和一次性创建事务。
2. 更新 OpenAPI 和前端 API client。
3. 实现团队列表筛选工具条和表格视觉。
4. 实现新建团队抽屉两步流程。
5. 补齐前后端测试。
6. 使用真实数据库做登录后烟测和截图验证。

## 11. 设计决策

- 新建团队使用右侧抽屉，不做独立页面。
- 初始成员只从已有租户 active 用户中选择。
- 创建团队在第二步一次性提交，服务端事务落库。
- 第一版不在抽屉里创建用户或邀请用户。
- 第一版不把团队详情 6 个 Tab 纳入本次重做范围。
