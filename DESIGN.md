# 设计风格

## 设计目标

前端 UI 采用企业级控制台设计风格：克制配色、清晰层级、稳定布局、高信息密度和可审计的操作反馈。视觉可以有科技感，但必须优先服务任务流、审批、运行状态、工件和审计等后台工作场景。

这份文件只定义通用视觉和交互规则，不定义具体业务内容模板。页面中的名称、指标、事件、状态、模型、时间线和说明文案必须来自当前业务需求、真实接口、已有 mock 约定或明确的占位数据说明。

## 核心原则

- **设计语言与业务内容分离**：视觉结构、色彩、阴影、卡片形态和交互样式可以复用；业务文案、指标数字、事件内容和状态含义必须按当前功能重新定义。
- **后台工具优先**：页面应保持清晰、可扫描、可操作的信息密度，避免营销页式大标题、大留白、装饰性堆叠和无意义动效。
- **状态可解释**：运行中、待确认、失败、受限、已完成、需人工处理等状态必须在视觉上可区分，并配有可追踪的原因、时间或下一步动作。
- **组件优先复用**：跨页面出现的视觉模式、状态标签、图标容器、空状态、错误提示和面板布局，应优先沉淀成复用组件。
- **渐进迁移**：旧页面不需要一次性重写；新增页面和重构页面应逐步向本设计语言靠拢。

## 视觉基调

### 当前目标风格：浅色液态玻璃控制台

SuperTeam Web 当前优先采用“近白浅底 + 冷色柔光 + 液态玻璃卡片 + 蓝绿色主强调 + 彩色语义图标”的视觉方向。整体应明亮、现代、轻量、有质感，但不能变成高饱和、强霓虹、营销页或纯装饰界面。

核心判断标准：

- 左侧导航以白色或近白色为主，叠加极淡青绿/淡蓝衬托，禁止回到大面积深色侧栏。
- 顶部栏保持清爽、轻量、玻璃质感，服务搜索、环境、团队、通知和账户操作。
- 主强调色使用低饱和蓝绿色，承担品牌、导航激活、主操作和焦点状态，不负责承载所有业务语义。
- 不同业务内容使用不同语义色：运行、成功、预警、危险、工件、审批、审计等应可一眼区分。
- 图标允许使用彩色或双色风格，但必须统一线宽、尺寸、饱和度和视觉重量，不能变成幼稚的彩色贴纸。
- 液态玻璃质感只用于 Shell、导航、顶部栏、选择器、卡片、轻量面板和悬浮层；表格、表单和高密度数据区域必须保持足够对比度。
- UI/UX 样式规范不携带业务内容模板。借鉴示例图时只能提取按钮、卡片、背景、阴影、边框、图标容器和动线节奏，不能把示例图里的业务模块、文案、指标或布局逻辑带入 SuperTeam。

### 色彩

- 主色用于主按钮、激活态、重点图标、焦点环和关键操作，不应在大面积背景中滥用。
- 背景采用白色、近白色、极淡青绿和极淡蓝色：页面底色偏冷，侧栏和顶部栏允许有玻璃感，但内容区域必须保持可读。
- 卡片和面板使用稳定底色、细边框和柔和阴影表达层级；半透明或玻璃质感只能作为轻量增强。
- 状态色必须保持语义一致：蓝色表示主操作或运行中，青色表示信息或 Runtime，绿色表示成功或健康，琥珀表示提醒或外部能力，红色表示失败或危险，紫色表示工件或智能工作项，灰蓝表示中性、审计或历史记录。
- 避免大面积单一色系、过重渐变、装饰性光斑和高饱和背景，整体应保持冷静、专业、可长时间使用。

推荐色彩基准：

| 用途 | 建议色值 | 使用范围 |
| --- | --- | --- |
| 品牌主色 | `#2CC7AA` | 主按钮、导航激活、焦点态、关键链接 |
| 品牌深色 | `#0A806F` | 主按钮渐变终点、强调文字、焦点边缘 |
| 主色浅背景 | `#E6FBF5` | 激活菜单、选中表格行、轻量提示背景 |
| Shell 淡色背景 | `#F8FBF7` / `#EEF8F4` | 页面底色、侧栏玻璃底色 |
| 信息 / Runtime | `#0891B2` | Runtime、信息提示、连接状态 |
| 成功 / 审批通过 | `#16A34A` | 成功、健康、已通过 |
| 预警 / 外部能力 | `#F59E0B` | 提醒、等待、外部能力调用 |
| 危险 / 风险 | `#EF4444` | 失败、危险、高风险、阻断 |
| 工件 / 智能工作项 | `#8B5CF6` | 工件、生成物、智能处理结果 |
| 决策 / 人类确认 | `#10B981` | 决策请求、确认、人工验收 |
| 中性 / 审计 | `#64748B` | 审计、历史、说明、弱状态 |

色彩比例建议：页面 70% 使用白色/近白色/淡青绿/淡蓝背景，20% 使用灰蓝文字与边框，10% 用于主色和语义色。一个页面中不要同时出现过多高饱和色块；语义色优先出现在图标、状态点、Badge、边框和小面积背景中。

### 液态玻璃拟态

液态玻璃拟态用于提升现代感和层次感，但必须保持企业后台的可读性和稳定感。

- 玻璃背景建议使用 `rgba(255, 255, 255, 0.72)` 到 `rgba(255, 255, 255, 0.86)`，叠加极淡蓝色背景。
- 模糊建议控制在 `backdrop-filter: blur(16px) saturate(1.15)` 到 `blur(24px) saturate(1.3)`。
- 边框建议使用 `rgba(156, 242, 218, 0.44)`、`rgba(191, 219, 254, 0.55)` 或 `rgba(255, 255, 255, 0.82)`，表达青绿/蓝白玻璃边缘。
- 阴影应柔和：优先使用 `0 16px 40px rgba(15, 23, 42, 0.06)` 这类低透明阴影，不使用厚重黑影。
- 顶部栏、左侧栏、工作区选择器、搜索框、菜单激活态、按钮、卡片和轻量浮层可以使用玻璃质感。
- 主按钮和激活菜单可以使用蓝绿色渐变：`#9CF2DA -> #2CC7AA -> #0A806F`。这组颜色用于品牌和导航强调，绿色成功色仍独立保留给健康/完成语义。
- 液态高光通过内阴影、细白边、低透明径向渐变和柔和投影形成；不要在页面中散落不受控的独立光斑。
- 主按钮、导航激活态和深色渐变承载文字或图标时，高光必须避开内容区域；禁止出现大块白色反光压住图标、中文文字或数字。
- 暗色主题必须使用独立的暗色玻璃覆盖：登录页 Shell、侧栏、卡片、图标容器、搜索框和顶部栏不能复用浅色主题的白色/浅绿色大面积高光。
- 表格正文、长表单、日志、审计明细和代码/错误内容不应使用强玻璃透明效果，避免降低阅读效率。

禁止事项：

- 不要用装饰性 orb、bokeh、强光斑来制造玻璃拟态。
- 不要让文本压在复杂渐变或透明背景上。
- 不要在同一区域叠三层以上半透明面板。
- 不要为了玻璃效果牺牲对比度、滚动性能和移动端可读性。

### 形态

- 圆角以当前主题 token 为准；卡片、按钮、输入框、菜单和弹窗应保持同一套圆角比例。
- 主按钮使用蓝绿色液态渐变和清晰 hover/active/focus 状态；辅助按钮使用 outline、secondary 或 ghost 风格。
- 图标优先使用现有图标库。工具按钮优先用图标表达，陌生图标需要 tooltip 或可访问标签。
- 关键图标可放入彩色圆角图标容器；普通列表、菜单和按钮图标可直接使用语义颜色。
- 标题区图标与内容区图标应区分层级：页面标题、面板标题、分组标题使用轻量小徽章；指标卡、能力项、列表项使用更强的彩色图标容器。
- 状态标签使用胶囊形态，搭配小圆点或图标表达状态。
- 页面区域之间用间距、面板和细边框区分，避免厚重分割线。

### 质感

- 质感服务层级，不作为页面主角。阴影、模糊、透明度和渐变必须克制使用。
- 需要强调浮层、弹窗、抽屉或导航吸顶时，可以使用轻量阴影、背景模糊和边框。
- 背景纹理只作为低对比辅助，不能影响内容阅读，也不能降低表格、图表和表单的辨识度。

## Shell 样式规范

Shell 是后续所有页面的视觉基准。新增功能页不得单独发明另一套侧栏、顶部栏、背景和选中态。

### 左侧导航

- 背景：白色或近白色玻璃底，带极淡青绿/淡蓝衬托；可以在顶部或边缘使用非常轻的冷色 wash，但不使用深色整栏。
- 宽度：默认保持当前 shadcn sidebar 体系，桌面展开态约 `236px` 到 `248px`；折叠态只保留图标和必要状态。
- 边界：右侧使用细蓝灰边框或玻璃高光，不使用厚重阴影。
- 分组：菜单按工作区、编排、执行、治理等领域分组。分组标题使用灰蓝色小字，降低视觉权重。
- 菜单项：高度约 `40px`，圆角不超过 `8px`，图标列对齐，文字左对齐，状态计数或状态点靠右。
- 激活态：蓝绿色液态胶囊或浅青绿玻璃背景 + 白色/深色文字 + 语义图标 + 柔和高光。激活态允许使用渐变，但必须限制在按钮/菜单项本身，不扩散成大面积色块。
- Hover：使用比激活态更淡的蓝色背景，不能改变布局尺寸。
- Footer：用户区保持紧凑，使用玻璃卡片或简洁分割线，不做厚重账户面板。

菜单图标建议：

| 菜单 | 图标色 | 语义 |
| --- | --- | --- |
| 工作台 | `#2CC7AA` | 主入口、全局概览 |
| 任务中心 | `#2685E8` 或 `#2CC7AA` | 任务和执行对象 |
| 数字员工 | `#14B8A6` | 角色、能力、员工配置 |
| 流程编排 | `#06B6D4` | 流程、节点、编排关系 |
| 外部能力 | `#F59E0B` | HTTP、连接器、外部系统 |
| 审批中心 | `#16A34A` | 人类确认、验收、通过 |
| Runtime 节点 | `#0EA5E9` | 运行节点、心跳、租约 |
| 权限中心 | `#6366F1` | 授权、策略、访问边界 |
| 审计日志 | `#64748B` | 历史、追踪、合规 |

图标实现规则：

- 优先继续使用 `lucide-react`，可以通过语义色、轻量背景容器或双色组合形成彩色图标风格。
- 同一层级图标尺寸保持一致；导航图标建议 `18px` 到 `20px`。
- 彩色图标不要使用阴影、强渐变或多色插画；颜色只表达领域分类。
- 高风险、失败、阻断等状态可以使用红色，但红色不能作为普通导航图标常态色。

### 顶部栏

- 顶部栏高度约 `64px`，白色玻璃底，底部使用淡蓝灰边框。
- 顶部栏只承载全局动作：折叠菜单、面包屑、全局搜索、环境、团队、通知、帮助、用户入口。
- 搜索框应是顶部视觉中心之一，使用白色或玻璃底、淡蓝边框和主色 focus ring。
- 环境和团队选择器使用轻量玻璃按钮，状态点使用语义色，例如生产环境使用绿色点。
- 通知、帮助、设置、头像等图标按钮保持轻量，不与主操作抢权重。
- 滚动吸顶时可以增加轻微背景模糊和阴影，但不要出现深色遮罩。

### 主内容背景

- 主内容区使用近白或极淡蓝底色，推荐 `#F8FBFF` 到 `#FFFFFF` 的冷色体系。
- 页面内容以白色面板、细边框和浅阴影建立层级；不要为了玻璃效果把所有业务卡片都做成半透明。
- 首屏应清晰呈现当前页面标题、主要操作、筛选/工具条和核心数据区。
- 页面间距要保持后台工具效率：舒适但不空旷，避免营销页式大留白。

### 内容语义色

新增页面必须按内容语义使用颜色，而不是按页面随意挑色。

| 语义 | 主色 | 背景色建议 | 使用对象 |
| --- | --- | --- | --- |
| 主操作 / 激活 | `#2CC7AA` | `#E6FBF5` | 主按钮、选中态、链接、焦点 |
| 运行中 / 处理中 | `#2685E8` | `#DBEAFE` | 任务运行中、队列、进度 |
| 信息 / Runtime | `#0891B2` | `#ECFEFF` | Runtime、连接、系统信息 |
| 成功 / 健康 | `#16A34A` | `#DCFCE7` | 成功、在线、健康、通过 |
| 决策 / 审批 | `#10B981` | `#D1FAE5` | 人类确认、验收、DecisionRequest |
| 预警 / 等待 | `#F59E0B` | `#FEF3C7` | SLA、待处理、注意事项 |
| 危险 / 阻断 | `#EF4444` | `#FEE2E2` | 失败、高风险、阻断、危险操作 |
| 工件 / 产物 | `#8B5CF6` | `#EDE9FE` | Artifact、报告、附件、输出 |
| 审计 / 中性 | `#64748B` | `#F1F5F9` | Audit、历史、说明、低优先级 |

状态色使用原则：

- Badge、状态点、边框和小面积背景优先；避免大面积纯色块。
- 关键状态不能只靠颜色表达，必须搭配文字、图标或结构。
- 同一语义全站保持一致，不在单页临时新增近似色。
- 表格行选中态使用主色浅背景，不改变行高和列宽。

## 主题 Token

- `--primary`：主操作、激活态、焦点状态和关键强调色。当前 Web 目标主色为低饱和蓝绿色体系。
- `--background`：页面底色，应偏白和极淡蓝，不使用深色作为默认工作区背景。
- `--foreground`：主要文字色。
- `--muted` / `--muted-foreground`：次级区域和辅助文字。
- `--card` / `--card-foreground`：卡片和面板底色。
- `--border` / `--input` / `--ring`：边框、输入框和焦点状态。
- `--sidebar` 系列：侧边栏及其状态。应表达浅色液态玻璃侧栏，而不是深色侧栏。
- `--chart-*`：图表、仪表盘和可视化辅助色，应与语义色保持相近而不是随机取色。
- `--radius`：基础圆角。

具体色值以当前前端主题文件为准。新增页面应优先使用语义 token，不直接堆叠原子颜色值来形成另一套视觉系统。

如需扩展项目级 token，优先在 `apps/web/src/styles/theme.css` 中补充语义变量，再通过 Tailwind token 使用。不要在单个页面内复制一组玻璃拟态 class。建议预留的项目级语义变量包括：

- `--superteam-shell-bg`：Shell 淡蓝背景。
- `--superteam-glass-bg`：玻璃面板背景。
- `--superteam-liquid-bg`：液态玻璃卡片背景。
- `--superteam-liquid-strong-bg`：主卡片、顶部栏、输入控件的更稳定玻璃背景。
- `--superteam-glass-border`：玻璃边框。
- `--superteam-glass-highlight`：玻璃内侧高光。
- `--superteam-shadow-low`、`--superteam-shadow-mid`、`--superteam-shadow-glow`：卡片、浮层、主操作阴影。
- `--superteam-menu-accent-soft`、`--superteam-menu-accent`、`--superteam-menu-accent-deep`：菜单和主操作蓝绿色渐变。
- `--superteam-sidebar-active`：侧栏激活背景。
- `--superteam-sidebar-hover`：侧栏 hover 背景。
- `--superteam-info`、`--superteam-success`、`--superteam-warning`、`--superteam-danger`、`--superteam-artifact`、`--superteam-decision`、`--superteam-neutral`：业务语义色。

已落地或建议沉淀的复用样式/组件：

- `LiquidShell` / `.superteam-auth-shell` / Shell 背景样式：登录页和控制台外层背景，负责近白底、冷色 wash、细网格和页面整体玻璃边界；是否需要 React 组件按页面复用情况再决定。
- `LiquidCard` / `.superteam-liquid-card`：通用卡片，负责半透明底、白色高光边、柔和阴影和稳定圆角。
- `LiquidPill` / `.superteam-liquid-pill`：搜索框、Tab 容器、次要胶囊按钮、轻量筛选器和非状态型短标签。
- `LiquidTabsList` / `LiquidTabsTrigger` / `.superteam-tabs-*`：页面级或模块级 Tab，使用浅色液态玻璃容器、玻璃胶囊激活态和底边流光选中锚点。
- `PrimaryLiquidButton` / `.superteam-primary-action`：主操作按钮，使用蓝绿色渐变和 glow，不用于失败、删除、危险动作。
- `SemanticIconTile` / `.superteam-icon-tile`：业务语义图标容器，用语义色表达任务、员工、Runtime、审批、审计等领域。
- `StatusBadge`：状态胶囊，支持状态点和语义色；用于运行、成功、预警、危险、工件、决策和审计等短状态。
- `MetricCard` / `.superteam-metric-card`：工作台指标卡，保持数据来自业务接口或明确 mock，不写入设计示例内容。

当前项目级设计组件统一放在 `apps/web/src/components/superteam/`，用于组合 shadcn/ui 基础组件和 SuperTeam 设计 token。`apps/web/src/components/ui/` 继续作为 shadcn primitive 层，除全站基础行为外，不承载业务组合组件。

新增组件时遵循以下规则：

1. 只要跨两个以上页面复用，优先沉淀到公共组件。
2. 只在单页使用的特殊布局，先保留在该 feature 内。
3. 不要修改 shadcn/ui 基础组件来承载业务视觉，除非是全站基础行为或项目已有定制。
4. 页面内 Tailwind class 应表达布局为主，复杂视觉样式优先下沉到复用组件或主题 token。

## 基础组件风格

以下规则定义通用视觉风格，不绑定具体业务模块。开发时优先使用现有组件和主题 token；只有在页面语义确实需要时，才做局部强化。

### 字体与层级

- 标题用于建立页面和面板层级，避免在后台页面中使用过大的展示型字号。
- 正文、说明文字、辅助元信息应形成清晰对比，但不要依赖过浅文字承载关键信息。
- 数字、状态、短 ID、时间等可扫描信息应保持稳定宽度和清晰对齐；长文本允许换行或截断，但必须能查看完整内容。
- 中文界面避免过密字重堆叠；同一区域内优先通过字号、字重、颜色和间距中的少量维度建立层级。

### 间距与密度

- 后台页面默认使用紧凑但不拥挤的密度，页面外边距、卡片内边距和列表行距应随内容复杂度适度放大。
- 同一页面内相同层级的面板、工具栏、表单组和列表项应保持一致间距。
- 高信息密度区域可以压缩留白，但不能牺牲点击目标、文字可读性和状态辨识度。
- 空状态、错误状态和确认区域可以保留更多呼吸感，用于降低误操作和理解成本。

### 按钮与操作控件

- 主按钮只承载当前区域最重要的提交或继续动作；同一区域不要并列多个同等视觉重量的主按钮。
- 次要操作使用 secondary、outline 或 ghost 风格；危险操作使用 destructive 风格，并避免与主按钮在视觉上混淆。
- 图标按钮用于工具型动作，必须提供可访问名称；不常见图标需要 tooltip。
- loading、disabled、focus-visible、hover 和 active 状态必须完整；异步动作期间避免出现可重复提交的视觉暗示。

### 表单控件

- 输入框、选择器、开关、单选、多选和文本域应保持统一高度、圆角、边框和焦点状态。
- 字段说明紧贴字段，错误信息紧贴出错位置；跨字段错误放在分组顶部或提交区附近。
- 只读态和禁用态要可区分：只读表示可查看不可编辑，禁用表示当前不可操作。
- 敏感信息、密钥和令牌类字段默认脱敏展示，复制、查看和重置动作需要明确状态反馈。

### 面板与卡片

- 面板用于组织页面结构，卡片用于承载独立信息单元；避免把大页面切成过多装饰性卡片。
- 面板标题区应包含标题、必要说明和少量操作；复杂操作应进入工具栏、菜单、抽屉或详情区。
- **高级卡片质感（Premium Cards）**：针对独立功能卡片（如团队卡片），若需在浅色背景上拉开物理层次，推荐采用“微渐变底色 + 内发光描边 + 悬浮动效”的组合范式：
  - **渐变基底**：摒弃纯白透明底，使用极微弱的对角线渐变（如 `bg-gradient-to-br from-background via-background/95 to-superteam-primary-soft/40`），确保在浅色环境下的体量感。
  - **折射高光**：配合细粒度的内侧边缘高光（如 `ring-1 ring-white/60`），增强拟真玻璃切割面的折射细节。
  - **呼吸反馈**：Hover 交互统一通过平滑的Z轴上浮（`hover:-translate-y-1`）搭配大半径的弥散光影（`hover:shadow-[var(--superteam-shadow-mid)]`）和光晕变色，提供极致的丝滑触感。
- 强调面板可以使用更明显的边框、图标或浅色底，但不应依赖重渐变或杂乱的大阴影。
- 列表卡片和表格行的 hover、selected、active 状态应有稳定层级，不能造成布局跳动。

### 标签与状态

- 标签用于表达短状态、分类或风险等级，文案应短而明确。
- 状态标签优先使用语义色和小图标；重要状态不能只靠颜色区分。
- 同一语义在全站保持一致颜色和形态，不为单个页面临时创造新的状态色。
- 标签数量要克制。多个状态同时出现时，应区分主状态和辅助标签，避免视觉噪音。

### 表格与列表

- 表头、行高、边框、hover 和选中态应保持克制，优先服务扫描、比较和批量操作。
- 操作列保持位置稳定；常用操作可直接显示，低频操作进入更多菜单。
- 空数据、加载中、加载失败和无权限状态应保留在数据区域内，不改变页面整体布局。
- 长字段、代码、路径、URL 和错误信息需要截断、换行或展开策略，不能挤压关键列。

### 浮层与反馈

- Dialog 用于确认、短表单和强打断信息；Drawer/Sheet 用于详情、编辑和长内容；Popover/Tooltip 用于轻量辅助说明。
- 浮层标题、内容、底部操作区应保持固定结构；危险确认必须突出影响范围。
- Toast 只承载短反馈，不替代页面内错误、表单校验和需要决策的信息。
- 遮罩、阴影和背景模糊应帮助聚焦，不应制造过强的舞台感。

### 图标

- 图标风格应统一，默认使用当前项目图标库和线性风格。
- 图标尺寸跟随按钮、标签、标题或列表项层级，不单独抢占视觉中心。
- 彩色图标容器用于强调关键对象或状态；普通导航、菜单和工具栏图标保持轻量。
- 纯装饰图标应隐藏给辅助技术；表达含义的图标必须配合文字、标签或可访问名称。

## 页面布局规则

### 后台页面

- 页面标题、操作按钮、筛选区、数据区应层次清晰，避免卡片套卡片。
- 页面首屏应直接呈现当前工作对象、关键状态和主要操作，不做营销式 hero。
- 工具栏、筛选项、批量操作、表格、详情抽屉和审计入口应保持位置稳定。

### 多 Tab / 子菜单页

- 同一业务域下的子菜单页应共享一个页面级页头和 Tab 容器，子页内部只保留面板标题和操作区。
- 页面级或模块级 Tab 优先使用 `LiquidTabsList` 和 `LiquidTabsTrigger`，不要在单页临时复制一套 Tab 样式。
- Tab 使用浅色液态玻璃容器，激活态使用玻璃胶囊面板、稳定边框和柔和阴影表达当前分组。
- 激活态底边可以使用蓝绿色流光作为选中锚点，但应贴近胶囊底部；不要把横线放在中文标签中部或让它成为主要装饰。
- Tab 标签需要适配中文长文案，容器应支持横向滚动或换行，不应压缩文字、改变行高或造成页面横向溢出。
- `apps/web/src/components/ui/tabs.tsx` 保持 shadcn/Radix primitive 层；SuperTeam 页面视觉应沉淀在 `apps/web/src/components/superteam/` 组合组件中。
- 子页切换不应重置与当前业务无关的全局状态。

### Dashboard / 控制台类页面

- 可以使用指标卡、事件流、资源用量、任务列表、拓扑/流程图等控制台模块。
- 指标数字和事件必须来自真实业务、接口响应或明确的 mock 约定。
- 拓扑图可以复用“节点卡 + 连线 + 网格背景”的视觉语言，但节点含义必须对应当前业务模型。
- 控制台模块应支持快速扫描：主指标突出，辅助指标收敛，异常状态可直接进入处理动作或详情。

### 列表 / 管理页

- 筛选区保持紧凑，优先横向工具条或轻量面板。
- 数据卡片和表格应使用统一面板、状态标签和图标容器。
- 列表卡片要有稳定的边框、背景、hover/active 层级；时间、状态、短 ID 等元信息默认不换行。
- 滚动容器内的首尾卡片应预留内边距，避免圆角、边框、ring 或阴影被裁剪。
- 批量操作、危险操作仍应遵循现有确认弹窗和权限规则。

### 表单 / 弹窗

- 表单以可读性和操作效率为优先，不强行加入装饰性渐变。
- 弹窗、抽屉、确认框使用现有组件组合，并在标题、按钮、状态提示上贴合控制台视觉。
- 表单标题、说明、字段、操作按钮之间应保持清晰层级；主操作使用主按钮，取消和次要操作使用 outline 或 ghost。
- 高风险操作必须使用明确确认流程，确认文案应说明影响范围、对象和不可逆后果。

## 状态与反馈规则

- 每个可操作页面至少覆盖 loading、empty、error、permission denied 和 disabled 状态。
- 长任务、异步执行、后台调度和外部调用必须展示当前阶段、最近更新时间、失败原因和下一步动作。
- Toast 只用于短反馈；需要用户阅读或决策的信息应放在页面状态、弹窗、抽屉或详情面板中。
- 表单校验错误应贴近字段展示；跨字段或业务规则错误应放在表单顶部或提交按钮附近。
- 可点击元素必须有 hover、active、focus-visible 和 disabled 状态；键盘可达性不能只依赖鼠标 hover。
- 危险、审批、权限不足、连接失败、执行失败等状态不能只靠颜色表达，必须配合图标、文案或结构变化。

## 移动端规则

- 移动端优先保证可读和可操作，允许将桌面多列布局降为单列。
- 顶部状态栏信息可折叠，保留导航触发、通知、用户入口等关键操作。
- 按钮文字不能溢出容器；长中文标题应自然换行或收缩布局。
- 面板之间保持足够垂直间距，不要为了保留桌面排布而造成移动端拥挤。
- 表格在移动端应优先转为卡片、横向滚动容器或字段折叠视图，不能直接压缩到不可读。

## 禁止事项

- 不要把与当前功能无关的任务、指标、事件、模型、平台状态等内容写入业务页面。
- 不要为了追求视觉效果而牺牲当前项目真实信息架构。
- 不要在每个页面复制一套独立的玻璃卡片和渐变按钮 class。
- 不要大面积使用装饰性光斑、orb、bokeh 或无意义插画。
- 不要把页面做成营销首页，后台页面必须服务于实际工作流。
- 不要用不可维护的绝对定位堆叠核心内容，除非是明确的图形/拓扑画布。
- 不要删除、重命名或替换公共工具函数和公共组件导出，除非同步更新全部调用点。
- 不要把其他项目的路径、模块名、角色名、指标名、接口名、截图验证工具名或业务状态原样带入当前项目。

## 实施检查清单

前端 UI 开发或改造完成前，至少检查：

- 是否只复用设计语言，业务内容是否来自当前功能的真实需求、接口或 mock 约定。
- 是否遵循基础组件风格，避免在单页临时创造另一套按钮、表单、标签、面板或浮层样式。
- 桌面和移动端是否无明显文字溢出、组件重叠、不可点击区域。
- 是否检查列表首尾边框、圆角、阴影没有被滚动容器裁剪。
- 是否覆盖 loading、empty、error、permission denied、disabled 和长任务执行中状态。
- 若引入或修改复用组件，是否用 `rg` 检查公共导出和调用点。
- 若页面存在报错，是否先修复路由懒加载、接口参数、公共工具导出等根因，再进行视觉改造。
- 是否按改动范围运行了 lint、必要测试或 build。
- 涉及可见页面时，是否用浏览器或端到端工具打开目标路由、切换关键 Tab，并检查无横向溢出。
