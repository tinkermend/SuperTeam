# 设计风格

## 设计目标

前端 UI 采用企业级控制台设计风格：克制配色、清晰层级、稳定布局、高信息密度和可审计的操作反馈。视觉可以有科技感，但必须优先服务任务流、审批、运行状态、工件和审计等后台工作场景。

这份文件只定义通用视觉和交互规则，不定义具体业务内容模板。页面中的名称、指标、事件、状态、模型、时间线和说明文案必须来自当前业务需求、真实接口、已有 mock 约定或明确的占位数据说明。

## 核心原则

- **设计语言与业务内容分离**：视觉结构、色彩、阴影、卡片形态和交互样式可以复用；业务文案、指标数字、事件内容和状态含义必须按当前功能重新定义。
- **后台工具优先**：页面应保持清晰、可扫描、可操作的信息密度，避免营销页式大标题、大留白、装饰性堆叠和无意义动效。
- **状态可解释**：运行中、待确认、失败、受限、已完成、需人工处理等状态必须在视觉上可区分，并配有可追踪的原因、时间或下一步动作。
- **组件优先复用**：跨页面出现的视觉模式、状态标签、图标容器、空状态、错误提示和面板布局，应优先沉淀成复用组件。
- **渐进迁移**：旧页面不需要一次性重写；新增页面和重构页面应逐步向本设计语言靠拢。

## 视觉基调

### 色彩

- 主色用于主按钮、激活态、重点图标和关键操作，不应在大面积背景中滥用。
- 背景应保持低对比和低干扰，允许使用浅色、深色或轻微层次背景，但内容区域必须保持可读。
- 卡片和面板使用稳定底色、细边框和柔和阴影表达层级；半透明或玻璃质感只能作为轻量增强。
- 状态色建议保持语义一致：蓝色表示信息或运行，绿色表示成功或健康，琥珀表示提醒，红色表示失败或危险，紫色表示智能、自动化或重点能力。
- 避免大面积单一色系、过重渐变、装饰性光斑和高饱和背景，整体应保持冷静、专业、可长时间使用。

### 形态

- 圆角以当前主题 token 为准；卡片、按钮、输入框、菜单和弹窗应保持同一套圆角比例。
- 主按钮使用主题主色和清晰 hover/active/focus 状态；辅助按钮使用 outline、secondary 或 ghost 风格。
- 图标优先使用现有图标库。工具按钮优先用图标表达，陌生图标需要 tooltip 或可访问标签。
- 关键图标可放入彩色圆角图标容器；普通列表、菜单和按钮图标可直接使用语义颜色。
- 标题区图标与内容区图标应区分层级：页面标题、面板标题、分组标题使用轻量小徽章；指标卡、能力项、列表项使用更强的彩色图标容器。
- 状态标签使用胶囊形态，搭配小圆点或图标表达状态。
- 页面区域之间用间距、面板和细边框区分，避免厚重分割线。

### 质感

- 质感服务层级，不作为页面主角。阴影、模糊、透明度和渐变必须克制使用。
- 需要强调浮层、弹窗、抽屉或导航吸顶时，可以使用轻量阴影、背景模糊和边框。
- 背景纹理只作为低对比辅助，不能影响内容阅读，也不能降低表格、图表和表单的辨识度。

## 主题 Token

- `--primary`：主操作、激活态和关键强调色。
- `--background`：页面底色。
- `--foreground`：主要文字色。
- `--muted` / `--muted-foreground`：次级区域和辅助文字。
- `--card` / `--card-foreground`：卡片和面板底色。
- `--border` / `--input` / `--ring`：边框、输入框和焦点状态。
- `--sidebar` 系列：侧边栏及其状态。
- `--chart-*`：图表、仪表盘和可视化辅助色。
- `--radius`：基础圆角。

具体色值以当前前端主题文件为准。新增页面应优先使用语义 token，不直接堆叠原子颜色值来形成另一套视觉系统。

新增组件时遵循以下规则：

1. 只要跨两个以上页面复用，优先沉淀到公共组件。
2. 只在单页使用的特殊布局，先保留在该 feature 内。
3. 不要修改 shadcn/ui 基础组件来承载业务视觉，除非是全站基础行为或项目已有定制。
4. 页面内 Tailwind class 应表达布局为主，复杂视觉样式优先下沉到复用组件或主题 token。

## 基础组件风格

以下规则定义通用视觉风格，不绑定具体业务模块。开发时优先使用现有组件和主题 token；只有在页面语义确实需要时，才做局部强化。

### 字体与层级

- 标题用于建立页面和面板层级，避免在后台页面中使用过大的展示型字号。
- 正文、说明文字、辅助元信息应形成清晰对比，但不要依赖过浅文字承载关键信息。
- 数字、状态、短 ID、时间等可扫描信息应保持稳定宽度和清晰对齐；长文本允许换行或截断，但必须能查看完整内容。
- 中文界面避免过密字重堆叠；同一区域内优先通过字号、字重、颜色和间距中的少量维度建立层级。

### 间距与密度

- 后台页面默认使用紧凑但不拥挤的密度，页面外边距、卡片内边距和列表行距应随内容复杂度适度放大。
- 同一页面内相同层级的面板、工具栏、表单组和列表项应保持一致间距。
- 高信息密度区域可以压缩留白，但不能牺牲点击目标、文字可读性和状态辨识度。
- 空状态、错误状态和确认区域可以保留更多呼吸感，用于降低误操作和理解成本。

### 按钮与操作控件

- 主按钮只承载当前区域最重要的提交或继续动作；同一区域不要并列多个同等视觉重量的主按钮。
- 次要操作使用 secondary、outline 或 ghost 风格；危险操作使用 destructive 风格，并避免与主按钮在视觉上混淆。
- 图标按钮用于工具型动作，必须提供可访问名称；不常见图标需要 tooltip。
- loading、disabled、focus-visible、hover 和 active 状态必须完整；异步动作期间避免出现可重复提交的视觉暗示。

### 表单控件

- 输入框、选择器、开关、单选、多选和文本域应保持统一高度、圆角、边框和焦点状态。
- 字段说明紧贴字段，错误信息紧贴出错位置；跨字段错误放在分组顶部或提交区附近。
- 只读态和禁用态要可区分：只读表示可查看不可编辑，禁用表示当前不可操作。
- 敏感信息、密钥和令牌类字段默认脱敏展示，复制、查看和重置动作需要明确状态反馈。

### 面板与卡片

- 面板用于组织页面结构，卡片用于承载独立信息单元；避免把大页面切成过多装饰性卡片。
- 面板标题区应包含标题、必要说明和少量操作；复杂操作应进入工具栏、菜单、抽屉或详情区。
- 强调面板可以使用更明显的边框、图标或浅色底，但不应依赖重渐变或大阴影。
- 列表卡片和表格行的 hover、selected、active 状态应有稳定层级，不能造成布局跳动。

### 标签与状态

- 标签用于表达短状态、分类或风险等级，文案应短而明确。
- 状态标签优先使用语义色和小图标；重要状态不能只靠颜色区分。
- 同一语义在全站保持一致颜色和形态，不为单个页面临时创造新的状态色。
- 标签数量要克制。多个状态同时出现时，应区分主状态和辅助标签，避免视觉噪音。

### 表格与列表

- 表头、行高、边框、hover 和选中态应保持克制，优先服务扫描、比较和批量操作。
- 操作列保持位置稳定；常用操作可直接显示，低频操作进入更多菜单。
- 空数据、加载中、加载失败和无权限状态应保留在数据区域内，不改变页面整体布局。
- 长字段、代码、路径、URL 和错误信息需要截断、换行或展开策略，不能挤压关键列。

### 浮层与反馈

- Dialog 用于确认、短表单和强打断信息；Drawer/Sheet 用于详情、编辑和长内容；Popover/Tooltip 用于轻量辅助说明。
- 浮层标题、内容、底部操作区应保持固定结构；危险确认必须突出影响范围。
- Toast 只承载短反馈，不替代页面内错误、表单校验和需要决策的信息。
- 遮罩、阴影和背景模糊应帮助聚焦，不应制造过强的舞台感。

### 图标

- 图标风格应统一，默认使用当前项目图标库和线性风格。
- 图标尺寸跟随按钮、标签、标题或列表项层级，不单独抢占视觉中心。
- 彩色图标容器用于强调关键对象或状态；普通导航、菜单和工具栏图标保持轻量。
- 纯装饰图标应隐藏给辅助技术；表达含义的图标必须配合文字、标签或可访问名称。

## 页面布局规则

### 后台页面

- 页面标题、操作按钮、筛选区、数据区应层次清晰，避免卡片套卡片。
- 页面首屏应直接呈现当前工作对象、关键状态和主要操作，不做营销式 hero。
- 工具栏、筛选项、批量操作、表格、详情抽屉和审计入口应保持位置稳定。

### 多 Tab / 子菜单页

- 同一业务域下的子菜单页应共享一个页面级页头和 Tab 容器，子页内部只保留面板标题和操作区。
- Tab 使用浅色玻璃容器，激活态用白底、主色文字和柔和阴影表达。
- 子页切换不应重置与当前业务无关的全局状态。

### Dashboard / 控制台类页面

- 可以使用指标卡、事件流、资源用量、任务列表、拓扑/流程图等控制台模块。
- 指标数字和事件必须来自真实业务、接口响应或明确的 mock 约定。
- 拓扑图可以复用“节点卡 + 连线 + 网格背景”的视觉语言，但节点含义必须对应当前业务模型。
- 控制台模块应支持快速扫描：主指标突出，辅助指标收敛，异常状态可直接进入处理动作或详情。

### 列表 / 管理页

- 筛选区保持紧凑，优先横向工具条或轻量面板。
- 数据卡片和表格应使用统一面板、状态标签和图标容器。
- 列表卡片要有稳定的边框、背景、hover/active 层级；时间、状态、短 ID 等元信息默认不换行。
- 滚动容器内的首尾卡片应预留内边距，避免圆角、边框、ring 或阴影被裁剪。
- 批量操作、危险操作仍应遵循现有确认弹窗和权限规则。

### 表单 / 弹窗

- 表单以可读性和操作效率为优先，不强行加入装饰性渐变。
- 弹窗、抽屉、确认框使用现有组件组合，并在标题、按钮、状态提示上贴合控制台视觉。
- 表单标题、说明、字段、操作按钮之间应保持清晰层级；主操作使用主按钮，取消和次要操作使用 outline 或 ghost。
- 高风险操作必须使用明确确认流程，确认文案应说明影响范围、对象和不可逆后果。

## 状态与反馈规则

- 每个可操作页面至少覆盖 loading、empty、error、permission denied 和 disabled 状态。
- 长任务、异步执行、后台调度和外部调用必须展示当前阶段、最近更新时间、失败原因和下一步动作。
- Toast 只用于短反馈；需要用户阅读或决策的信息应放在页面状态、弹窗、抽屉或详情面板中。
- 表单校验错误应贴近字段展示；跨字段或业务规则错误应放在表单顶部或提交按钮附近。
- 可点击元素必须有 hover、active、focus-visible 和 disabled 状态；键盘可达性不能只依赖鼠标 hover。
- 危险、审批、权限不足、连接失败、执行失败等状态不能只靠颜色表达，必须配合图标、文案或结构变化。

## 移动端规则

- 移动端优先保证可读和可操作，允许将桌面多列布局降为单列。
- 顶部状态栏信息可折叠，保留导航触发、通知、用户入口等关键操作。
- 按钮文字不能溢出容器；长中文标题应自然换行或收缩布局。
- 面板之间保持足够垂直间距，不要为了保留桌面排布而造成移动端拥挤。
- 表格在移动端应优先转为卡片、横向滚动容器或字段折叠视图，不能直接压缩到不可读。

## 禁止事项

- 不要把与当前功能无关的任务、指标、事件、模型、平台状态等内容写入业务页面。
- 不要为了追求视觉效果而牺牲当前项目真实信息架构。
- 不要在每个页面复制一套独立的玻璃卡片和渐变按钮 class。
- 不要大面积使用装饰性光斑、orb、bokeh 或无意义插画。
- 不要把页面做成营销首页，后台页面必须服务于实际工作流。
- 不要用不可维护的绝对定位堆叠核心内容，除非是明确的图形/拓扑画布。
- 不要删除、重命名或替换公共工具函数和公共组件导出，除非同步更新全部调用点。
- 不要把其他项目的路径、模块名、角色名、指标名、接口名、截图验证工具名或业务状态原样带入当前项目。

## 实施检查清单

前端 UI 开发或改造完成前，至少检查：

- 是否只复用设计语言，业务内容是否来自当前功能的真实需求、接口或 mock 约定。
- 是否遵循基础组件风格，避免在单页临时创造另一套按钮、表单、标签、面板或浮层样式。
- 桌面和移动端是否无明显文字溢出、组件重叠、不可点击区域。
- 是否检查列表首尾边框、圆角、阴影没有被滚动容器裁剪。
- 是否覆盖 loading、empty、error、permission denied、disabled 和长任务执行中状态。
- 若引入或修改复用组件，是否用 `rg` 检查公共导出和调用点。
- 若页面存在报错，是否先修复路由懒加载、接口参数、公共工具导出等根因，再进行视觉改造。
- 是否按改动范围运行了 lint、必要测试或 build。
- 涉及可见页面时，是否用浏览器或端到端工具打开目标路由、切换关键 Tab，并检查无横向溢出。
